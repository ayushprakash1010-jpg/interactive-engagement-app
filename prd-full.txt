Internal Admin Console – Detailed Implementation Plan for IEP
1. Overview

As the Interactive Engagement Platform (IEP) grows and starts supporting more users and organizations, managing customer accounts, events, integrations, and technical issues directly through the database will become difficult and risky.

We propose building a separate Internal Admin Console for the IEP team.

The existing platform will effectively have three interfaces:

IEP Platform
│
├── Participant Experience
│   └── Join events and participate in Polls, Quizzes,
│       Q&A, Surveys, Feedback and Word Clouds
│
├── Host Dashboard
│   └── Create and manage events, activities,
│       analytics, AI Studio and integrations
│
└── Internal Admin Console
    └── Manage and support the entire IEP platform

The Admin Console will only be accessible to authorized internal team members.

The primary goal is to allow the support and operations teams to:

Find and troubleshoot customer accounts.
Monitor events.
Investigate integration problems.
Manage organizations and users.
View platform-wide analytics.
Investigate errors and support issues.
Monitor platform and integration health.
Perform controlled administrative actions.
Maintain an audit trail of sensitive operations.

This removes the need for support staff to directly access MongoDB or ask developers to investigate routine customer issues.

3. Authentication and Authorization

Security should be the first implementation priority because the Admin Console will have access to sensitive platform-wide information.

We should define multiple roles.

Platform Admin
    │
    ├── Full platform access
    │
    ├── Manage users
    ├── Manage organizations
    ├── Manage events
    ├── Change feature flags
    └── Administrative actions


Support Agent
    │
    ├── Search customers
    ├── View events
    ├── View integration status
    └── Troubleshoot issues


Operations
    │
    ├── Platform analytics
    ├── System health
    └── Usage monitoring


Customer Success
    │
    ├── Organizations
    ├── Plans
    ├── Usage
    └── Account information

Initially, we can start with fewer roles, such as:

host
admin

Later, the permission system can evolve into:

host
support
operations
admin
super_admin

Every Admin API request should pass through:

Request
   ↓
JWT Validation
   ↓
Authentication Guard
   ↓
Role/Permission Guard
   ↓
Admin API
   ↓
Service
   ↓
Database

Simply hiding admin pages in the frontend is not enough. Authorization must always be enforced by the NestJS backend.

15. Impersonation – Implement Carefully

The original proposal includes:

"Log in as this user."

This is useful but also one of the highest-risk features in the entire Admin Console.

I would not include it in the MVP.

Initially, support agents should be able to inspect:

User

Events

Configuration

Integrations

Usage

Audit History

If impersonation becomes necessary later, it should use a temporary, restricted support session.

For example:

Support Agent

↓

Request Impersonation

↓

Provide Reason

↓

Create Temporary Session

↓

Read-Only by Default

↓

Session Automatically Expires

↓

Audit Log Created

This prevents support staff from silently performing actions as customers.

18. Key Decisions Before Implementation

Before development starts, I recommend resolving four architectural decisions.

Authentication: Decide whether internal admins use the existing Auth0 tenant or a dedicated employee authentication system.

Authorization: Decide whether the initial model is simply host/admin or a more granular permissions model.

Admin UI: Decide whether to reuse the existing IEP design system or use an admin-oriented framework. Since you already have a developed design system, I would initially reuse it to maintain consistency and avoid introducing another major dependency.

Organizations: Determine whether the current data model already supports multi-tenancy properly. If not, organization architecture should be designed before building organization management.

Final Proposed Direction

The Admin Console should not simply be another dashboard. It should become the internal client-support and operations layer of IEP.
